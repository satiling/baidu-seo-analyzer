Node.js内存泄漏排查实战：从502错误到根因定位的完整过程

作者：张三 | 2026-06-15

上周我在部署一个Node.js项目时踩了个大坑，花了整整3个小时才定位到问题。这里记录下完整的排查过程，希望对大家有帮助。这次踩坑让我意识到V8的GC机制有些边界情况需要特别注意。

问题现象

事情是这样的：我们在AWS EC2 t3.medium（4GB内存）上用PM2跑Node服务，上线2周一直正常，但昨天突然报502错误。我第一时间看了下PM2日志，发现是内存溢出（OOM）导致进程被kill。

具体表现：

  • PM2日志显示"Script execution timed out"
  • 服务器free内存从正常的2.1GB降到200MB
  • Node进程内存占用从800MB涨到2.8GB
  • 用户访问间歇性502

排查步骤

具体操作步骤如下，建议按顺序执行：

第一步：确认内存状况

用free -h查看内存，发现4GB的实例只剩200MB可用。再用top -p $(pgrep node)命令发现有一个node进程占了2.8GB内存，明显高于正常水平。

第二步：heap snapshot对比

用node --inspect server.js参数启动Node，连Chrome DevTools做heap snapshot对比。具体做法是：先拍一个快照，等待10分钟后再拍一个，对比两个快照的差异。

对比结果发现：一个定时器回调里创建的大数组（每次约50MB）持续增长但从未被清理。根据V8官方文档，闭包引用的数组不会被GC自动回收。

第三步：定位根因

问题代码大致如下：

setInterval(() => {
  const data = fetchLargeData(); // 每次约50MB
  cache.push(data); // cache是闭包外的数组，持续增长
}, 5 * 60 * 1000); // 每5分钟一次

问题很明显：cache数组在闭包外定义，每5分钟push一次数据但从不清理。2周累计下来就是2.8GB。

修复方案

修复方案很简单，在定时器回调里加了cache.length = 0清空数组，或者限制数组长度：

setInterval(() => {
  const data = fetchLargeData();
  cache.push(data);
  if (cache.length > 10) cache.shift(); // 只保留最近10条
}, 5 * 60 * 1000);

上线后内存稳定在800MB左右，问题解决。根据我们的监控数据，修复前内存增长速率是200MB/天，修复后稳定不增长。

经验总结

这次踩坑让我意识到几个要点：

  1. V8的GC对闭包引用的数组不会自动回收，特别是定时器里的引用
  2. 建议大家在用setInterval时务必注意闭包内的变量清理
  3. 生产环境务必开启--max-old-space-size限制
  4. 定期做heap snapshot监控，避免内存缓慢泄漏

根据V8官方文档，Node.js 18以上的内存管理有改进，但闭包引用依然是内存泄漏的高发点。我们团队后续打算引入heapdump做定期快照监控，避免再出现类似问题。

不同于网上很多文章说的"GC会自动处理"，实际上闭包引用是GC的盲区。相比之下，Java的WeakReference能更好地处理这类场景。
